## Cross-Phase Issues

### BLOCKER: Admin Retry Drops OpenAI Style-Copy Continuation Metadata

Affected requirements: `OPS-03`, `OED-02`, `OED-03`

Phase 10 stores and requires `copyTarget` and `selectedImageId` for OpenAI style-copy continuation. `GenerationService.create()` persists those fields into `promptData` and enqueues them for normal user flow. The worker then verifies queued fields against persisted fields before calling a provider runtime.

The Phase 7 admin retry path requeues only `styleReferenceId`, source/character/texture paths, prompt, provider, providerModel, and options. It does not pass persisted `promptData.copyTarget` or `promptData.selectedImageId`.

Evidence:

- Required by worker guard: `apps/api/src/worker.ts:112`, `apps/api/src/worker.ts:117`, `apps/api/src/worker.ts:449`.
- Written by Phase 10 create flow: `apps/api/src/services/generation.service.ts:302`, `apps/api/src/services/generation.service.ts:303`, `apps/api/src/services/generation.service.ts:340`, `apps/api/src/services/generation.service.ts:342`.
- Omitted by admin retry enqueue: `apps/api/src/services/admin.service.ts:569`.
- Admin retry is user-accessible from admin UI/API: `apps/api/src/routes/admin/generations.routes.ts:35`, `apps/web/src/components/admin/generation-table.tsx:155`.

Impact:

- A failed OpenAI style-copy generation can be created normally, but admin retry requeues an incomplete job.
- The retry will fail before vendor calls with a queue/stored metadata mismatch, so the admin recovery flow cannot complete for OpenAI style-copy follow-ups.
- Existing automated retry tests cover normal Gemini retry and OpenAI Sketch retry, but not OpenAI style-copy retry.

Suggested fix:

- Include `copyTarget: stringValue(promptData.copyTarget)` and `selectedImageId: stringValue(promptData.selectedImageId)` in `AdminService.retryGeneration()` job payload.
- Add a regression test for failed OpenAI style-copy retry.

### WARNING: Phase 08 Transparent-Background Statement Is Stale For IP Change v2

Affected requirements: `OIP-02` only as documentation/expectation risk

Phase 08 verification says the transparent-background checkbox is carried through as an option, but the current OpenAI IP Change v2 page has no transparent-background control and the API rejects `provider=openai`, `mode=ip_change`, `transparentBackground=true`.

Evidence:

- No `transparent` UI in `apps/web/src/app/projects/[id]/ip-change/openai/page.tsx`.
- API/service validation rejects OpenAI IP transparent background: `apps/api/src/routes/generation.routes.ts:99`, `apps/api/src/services/generation.service.ts:175`.
- Phase 08 verification stale claim: `.planning/phases/08-openai-ip-change-parity/08-VERIFICATION.md`.

Impact:

- Not a blocker for the listed v1.1 IP Change requirements because `OIP-02` asks for structure/viewpoint/background/hardware preservation options, not transparent output.
- It is a release-note/documentation mismatch and should not be counted as verified transparent parity.

### WARNING: Responses API Top-Level Model Is Hardcoded Outside Config

Affected requirements: `OED-02`, `OED-03`

`OpenAIImageService.generateStyleCopyWithLinkage()` defaults `OPENAI_RESPONSES_IMAGE_MODEL` to `gpt-5.5`. Project skill references say the Responses API top-level model must be a text-capable mainline model, with `gpt-5.4` as the documented example. This may be fine if `gpt-5.5` is available in the target account, but it is not configured or documented in app config.

Evidence:

- Default: `apps/api/src/services/openai-image.service.ts:415`.
- Project reference: `.codex/skills/mockup-openai-image-runtime/references/node-runtime.md`.

Impact:

- Live OpenAI style-copy linkage remains unverified. If the default model is unavailable, style-copy with stored linkage can fail before falling back unless the operator sets `OPENAI_RESPONSES_IMAGE_MODEL`.
