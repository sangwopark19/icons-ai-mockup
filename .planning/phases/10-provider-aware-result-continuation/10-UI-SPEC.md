# Phase 10 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — use the existing manual Next.js/Tailwind product UI |
| Preset | not applicable |
| Component library | local React components (`Button`, `Input`, `ImageUploader`) plus existing result panels, history cards, and modal patterns |
| Icon library | `lucide-react` for new utility/action icons where available; existing workflow emoji may remain where already used |
| Font | `Pretendard Variable`, `Pretendard`, system sans |

Source: `components.json` is absent, `apps/web/src/app/globals.css` defines the token system, and Phase 8/9 UI-SPECs locked OpenAI workflow surfaces to the existing manual Tailwind UI. Do not initialize shadcn or add registry blocks in this phase.

User-facing product screens use `v1` and `v2` as the provider/model signal. Do not show raw `Gemini`, `OpenAI`, `GPT Image 2`, `gpt-image-2`, `provider`, or `providerModel` labels in normal project, result, history, edit, regenerate, or style-copy UI. Raw provider/model values remain available only in API data, admin/ops/debug surfaces, tests, and logs.

---

## Phase UI Scope

| Surface | Contract |
|---------|----------|
| `/projects/:id/generations/:genId` | Keep the existing result page layout. Every generation must show a `v1`/`v2` badge derived from persisted `generation.provider`, not route state or query params. Enable v2 follow-up actions for partial edit, style copy, and same-condition regeneration. |
| Result action stack | Replace the Phase 8/9 disabled v2 follow-up copy with working controls. Button order stays: `부분 수정`, `히스토리에 저장`, `스타일 복사 (IP 변경)`, `스타일 복사 (새 제품 적용)`, `동일 조건 재생성`, `조건 수정`. |
| Partial edit modal | Reuse the existing freeform `부분 수정` modal for both v1 and v2. For v2, submit against the originating OpenAI provider with the selected result image as input and navigate to the new one-result generation page. |
| Same-condition regeneration | Use the existing `동일 조건 재생성` action. For v2, create a fresh OpenAI request that replays the original provider, model, stored inputs, prompt, and options, then navigate to a new two-candidate result. |
| OpenAI style-copy page | Add a dedicated route: `/projects/:id/style-copy/openai?styleRef=:generationId&copyTarget=ip-change\|new-product&imageId=:selectedImageId`. The page must show the approved output as the style reference and collect the new target asset before submission. |
| `/projects/:id/history` | Every history card must show the workflow label plus `v1`/`v2` badge derived from persisted provider data. Reopening a card must preserve the provider-pinned result behavior. |
| Admin/ops surfaces | Raw `provider`, `providerModel`, OpenAI request IDs, response IDs, image-call IDs, revised prompts, and provider trace metadata may remain visible only in operational surfaces already designed for debugging. |

## Version Badge Contract

| Persisted provider/model | User-facing badge | Normal product UI text |
|--------------------------|-------------------|------------------------|
| `provider = "gemini"` | `v1` | Show only `v1`; do not show `Gemini` or raw model name. |
| `provider = "openai"` | `v2` | Show only `v2`; do not show `OpenAI`, `GPT Image 2`, or `gpt-image-2`. |
| Missing legacy provider after API defaulting | `v1` | Treat as legacy v1 only if the API record has already defaulted provider/model. |
| Unknown or unsupported provider | No normal action state | Show a clear failure state and block follow-up actions; do not guess a provider from badge text. |

Badge placement:

| Surface | Placement |
|---------|-----------|
| Result header | Beside `생성 결과` in the existing header title row. |
| Candidate/action panel | The count label remains `생성된 이미지 (2개)` for v2 two-candidate flows; partial-edit v2 result uses `수정 결과`. |
| History card | In the metadata row immediately after workflow label, matching Phase 8/9 badge treatment. |
| Style-copy page | On the style reference preview header as `스타일 기준 · v2`. |

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge inner gaps, checkbox-to-label fine spacing |
| sm | 8px | Compact row gaps, modal button gap, thumbnail gaps, inline metadata gap |
| md | 16px | Default element spacing, card grid gap, modal body spacing, header horizontal padding |
| lg | 24px | Result action panel padding, style-copy panel padding, modal padding |
| xl | 32px | Page vertical rhythm, result layout gap, style-reference to target-form spacing |
| 2xl | 48px | Empty state vertical padding, full-page loading spacing |
| 3xl | 64px | Page-level spacing only for full empty/loading states |

Exceptions: existing product cards and result image frames may keep `rounded-xl` to match current project pages. New compact controls use `rounded-lg`. Icon-only controls must be at least `40px` on desktop and `44px` on mobile. Candidate thumbnails, style-reference preview, and target upload previews must use fixed aspect ratios so badges, overlays, and loading states do not shift layout.

## Typography

Use exactly these sizes and these two weights for new Phase 10 UI.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 12px | 600 | 1.4 |
| Body | 14px | 400 | 1.5 |
| Heading | 18px | 600 | 1.2 |
| Display | 24px | 600 | 1.2 |

Labels include `v1`/`v2` badges, field labels, candidate labels, helper text, validation text, and style-reference metadata. Body text includes modal descriptions, result action copy, empty states, and inline success/error messages. Heading is for page titles, modal titles, and panel titles. Display is for the project name only if the surrounding project layout already uses it.

## Interaction Contract

| Interaction | Required behavior |
|-------------|-------------------|
| Provider source of truth | All result/history/follow-up actions derive provider and model from the persisted generation record. Do not infer runtime from current route, badge text, query params, selected tab, or client-side state alone. |
| Version display | Every result and history item with provider data shows `v1` or `v2` text. The label must be visible text, not color-only. |
| Raw metadata boundary | Product UI hides raw provider/model names. Admin/ops/debug surfaces may show raw metadata for support, including OpenAI request/response/image-call IDs. |
| V2 follow-up enablement | Remove Phase 8/9 disabled v2 follow-up behavior. For completed OpenAI v2 generations with a selected image, `부분 수정`, both style-copy actions, and `동일 조건 재생성` are enabled. |
| Unsupported state | If selected image, provider, model, or required stored inputs are missing, disable only the affected action and show the specific error copy above. Never fall back to Gemini or silently run v1. |
| Partial edit open | Clicking `부분 수정` opens the existing modal. The modal starts empty every time and keeps focus inside the dialog until cancelled or submitted. |
| Partial edit submit | Disable submit until `수정 요청` has non-whitespace text and a selected image exists. Submit creates a new generation/result record and routes to `/projects/:id/generations/:newGenId`. |
| Partial edit output | OpenAI v2 partial edit returns exactly one result image. The result page must not render two empty candidate slots for partial edits. |
| Partial edit preservation | Modal copy must communicate surgical edit behavior: only the requested target changes; product body, camera angle, crop, background, lighting, labels, hardware, and non-target details remain preserved. |
| Same-condition regeneration | Clicking `동일 조건 재생성` sends no editable form data. It reuses the original provider, provider model, stored input assets, prompt, and options from the source generation. |
| V2 regeneration output | OpenAI v2 regeneration returns exactly two candidates and uses the existing candidate selection, save, reopen, and download lifecycle. |
| Regeneration missing inputs | If stored inputs are missing or invalid, show `원본 입력을 확인할 수 없어 동일 조건 재생성을 시작하지 못했습니다.` and stay on the current result page. |
| Gemini follow-up isolation | Existing v1 edit, regeneration, and style-copy behavior remains available and must continue using Gemini-only state such as `thoughtSignature` where applicable. |
| OpenAI follow-up isolation | v2 partial edit, style copy, and regeneration must not read or send Gemini `thoughtSignature`. OpenAI lineage uses OpenAI response/image linkage where available. |
| Style-copy routing | For v2 results, both style-copy buttons route to `/projects/:id/style-copy/openai` with `styleRef`, `copyTarget`, and `imageId` query params. For v1 results, preserve the existing Gemini style-copy route behavior unless implementation intentionally refactors it without changing behavior. |
| Style reference display | The style-copy page must show the approved output image above the target upload on mobile and in a left panel on desktop. It must include `스타일 기준 · v2` and helper copy near the image. |
| Style target collection | `copyTarget=ip-change` requires a new character/IP image. `copyTarget=new-product` requires a new product image. Do not ask the user to choose provider or model on this page. |
| Style-copy fallback | If OpenAI response/image linkage is missing, use the selected result image as the style reference. Do not block the user or surface raw linkage details unless the backend cannot access the image file. |
| Style-copy output | OpenAI v2 style copy returns exactly two candidates and routes to the existing result page. Candidate labels, selection, save, reopen, and download match Phase 8/9 v2 behavior. |
| Condition edit | `조건 수정` remains a route-back action. For v2 `ip_change`, route to `/projects/:id/ip-change/openai`; for v2 `sketch_to_real`, route to `/projects/:id/sketch-to-real/openai`; for v1, preserve existing routes. |
| Loading states | Result page polling keeps the existing 2-second pattern. Button-level follow-up requests show loading in the initiating button and prevent duplicate submissions. |
| Save lifecycle | `히스토리에 저장` saves the currently selected image. Show success/error message near the action stack for 3 seconds; do not navigate away automatically. |
| Download lifecycle | `선택 이미지 다운로드` downloads the selected image only. If no candidate/result is selected, the control is disabled rather than downloading the first image silently. |
| History reopen | History cards always reopen `/projects/:id/generations/:genId`; the result page derives version and follow-up routing from persisted provider data after fetch. |
| Delete history | Delete action remains destructive and requires confirmation. Deleting one card removes it locally after successful API response and must not affect other generations. |

## Responsive Behavior

| Breakpoint | Contract |
|------------|----------|
| Mobile `< 640px` | Result page stacks selected image above candidate/action panel. Style-copy page stacks style reference first, then target uploader, instructions, and CTA. Buttons use full width and minimum 44px height. |
| Tablet `640px-1023px` | Result page may remain stacked. History grid uses two or three columns where space allows. Style-copy page may use a two-column grid only if both preview and upload controls remain readable. |
| Desktop `>= 1024px` | Result page keeps `1fr / 300px` or equivalent. Style-copy page uses `minmax(0, 1fr) / 360px` or equivalent with style reference left and target/action panel right. |

No text may overflow buttons, badges, cards, modal, upload zones, or thumbnails at 360px width. Long project names, prompt text, and target filenames wrap. Provider/model internals remain hidden on product screens.

## Accessibility Contract

| Area | Requirement |
|------|-------------|
| Version labels | `v1`/`v2` badges must appear as visible text and remain in the accessible name or nearby text context. |
| Follow-up buttons | Buttons are keyboard reachable, expose loading/disabled state, and keep stable labels while loading. Use `aria-describedby` for action-specific error/help text when disabled. |
| Partial edit modal | Use dialog semantics or existing modal behavior with focus trap, Escape/Cancel dismissal, labelled title, and focus return to the `부분 수정` button after close. |
| Partial edit input | The input must be associated with `수정 요청`; empty validation text must be announced or placed adjacent to the field. |
| Candidate thumbnails | Buttons need `aria-label` values: `후보 1 선택`, `후보 2 선택`. Selected candidate exposes visible `선택됨` text and `aria-pressed=true` or equivalent selected state. |
| One-result edit output | One-result partial edit pages should not expose fake candidate buttons. The selected image alt text is `선택된 v2 부분 수정 결과`. |
| Style reference image | Use alt text `스타일 기준 v2 결과 이미지`. The style reference panel must not rely on border color alone; include `스타일 기준 · v2` text. |
| Style target upload | Preserve existing paste/file-select keyboard access. Remove buttons need Korean `aria-label` values: `새 캐릭터 이미지 제거` or `새 제품 이미지 제거`. |
| History cards | Card link accessible name includes workflow label, version badge, and date context. Delete button has `aria-label="히스토리 삭제"` and does not trigger card navigation. |
| Focus | Use existing `focus-visible` brand ring. Do not remove outlines from links, modal controls, upload controls, candidate thumbnails, history cards, or icon buttons. |
| Raw metadata | Do not add raw provider/model text only for screen readers in product UI; it violates the same product-label boundary. Use `v1`/`v2` consistently. |
