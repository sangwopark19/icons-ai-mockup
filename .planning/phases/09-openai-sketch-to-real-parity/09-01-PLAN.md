# Plan 09-01: Backend OpenAI Sketch To Real Runtime

<objective>
Unblock OpenAI `provider = "openai"` and `providerModel = "gpt-image-2"` jobs for `mode = "sketch_to_real"`. The backend must accept a required sketch image, optional texture/material reference, product/material guidance, preservation defaults, two candidate outputs, OpenAI support metadata, and transparent-background fulfillment through a local Sharp-based uniform light-background post-process rather than unsupported GPT Image 2 parameters.
</objective>

<scope_note>
09-01 intentionally keeps backend request contracts, OpenAI runtime prompt construction, worker dispatch, transparent post-processing, and schema-state verification in one backend runtime plan. Transparent-background behavior is cohesive with this plan because the `generateSketchToReal()` prompt must create the clean uniform light/near-white opaque background precondition, `worker.ts` must fail closed when post-processing quality gates fail, `generation.service.ts` must persist `hasTransparency`, and candidate output ordering must remain stable through the same backend path. Splitting the transparent post-process into another backend plan would create overlapping ownership of `apps/api/src/services/generation.service.ts` and `apps/api/src/worker.ts` and force dependency churn into 09-03/09-04 without reducing implementation ambiguity. UI entry, result/history lifecycle, and smoke evidence remain separated in 09-02, 09-03, and 09-04.

Scope exception: this plan intentionally exceeds the usual task/file warning threshold because backend contract, OpenAI runtime, worker dispatch, upload persistence, and transparent post-process form one atomic provider/runtime contract. Wave 1 depends on this single backend contract being internally consistent before the independent form plan (09-02) and downstream result/smoke plans can rely on provider/model validation, two-candidate ordering, OpenAI request metadata, transparent-background post-processing, and generated-image persistence semantics.
</scope_note>

<threat_model>
## Threats

| ID | Severity | Threat | Mitigation |
|----|----------|--------|------------|
| T-09-01 | high | OpenAI Sketch jobs route into Gemini or a mismatched model | Preserve worker persisted-vs-queued `provider` and `providerModel` validation before dispatch. |
| T-09-02 | high | User-supplied sketch or texture paths escape the owner/project upload scope | Reuse `validateGenerationImagePaths()` with project upload prefixes for `sourceImagePath` and `textureImagePath`. |
| T-09-03 | high | Raw OpenAI API key, image bytes, or raw vendor body leaks to logs or product UI | Use provider-scoped active key lookup; persist only support IDs, revised prompt, and providerTrace. |
| T-09-04 | medium | GPT Image 2 receives unsupported `background: "transparent"` or `input_fidelity` | Unit tests assert neither field is present in `client.images.edit()` calls. |
| T-09-05 | medium | Texture reference changes product shape, scene, logos, text, or props | Prompt names Image 2 as material/finish/color behavior only and repeats hard constraints. |
| T-09-06 | medium | Transparent-background requests silently save opaque or visually poor outputs | Run local post-process after opaque generation; set `hasTransparency = true` only when alpha and visual-quality checks pass; fail generation if confidence or quality is below threshold. |
</threat_model>
