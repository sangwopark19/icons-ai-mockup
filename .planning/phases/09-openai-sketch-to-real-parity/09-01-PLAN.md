# Plan 09-01: Backend OpenAI Sketch To Real Runtime

<objective>
Unblock OpenAI `provider = "openai"` and `providerModel = "gpt-image-2"` jobs for `mode = "sketch_to_real"`. The backend must accept a required sketch image, optional texture/material reference, product/material guidance, preservation defaults, two candidate outputs, OpenAI support metadata, and transparent-background fulfillment through post-processing rather than unsupported GPT Image 2 parameters.
</objective>

<threat_model>
## Threats

| ID | Severity | Threat | Mitigation |
|----|----------|--------|------------|
| T-09-01 | high | OpenAI Sketch jobs route into Gemini or a mismatched model | Preserve worker persisted-vs-queued `provider` and `providerModel` validation before dispatch. |
| T-09-02 | high | User-supplied sketch or texture paths escape the owner/project upload scope | Reuse `validateGenerationImagePaths()` with project upload prefixes for `sourceImagePath` and `textureImagePath`. |
| T-09-03 | high | Raw OpenAI API key, image bytes, or raw vendor body leaks to logs or product UI | Use provider-scoped active key lookup; persist only support IDs, revised prompt, and providerTrace. |
| T-09-04 | medium | GPT Image 2 receives unsupported `background: "transparent"` or `input_fidelity` | Unit tests assert neither field is present in `client.images.edit()` calls. |
| T-09-05 | medium | Texture reference changes product shape, scene, logos, text, or props | Prompt names Image 2 as material/finish/color behavior only and repeats hard constraints. |
| T-09-06 | medium | Transparent-background requests silently save opaque outputs | Run post-process after opaque generation and set `hasTransparency = true` only for alpha PNG outputs; fail generation if post-process fails. |
</threat_model>
